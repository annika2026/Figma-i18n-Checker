# 文本检测缓存池功能说明

## 功能概述

在文本宽度检测过程中，为了避免对重复的文本进行多次检测，我们实现了一个缓存池功能。该功能可以显著提高检测效率，特别是在存在大量重复翻译文本的场景下。

## 工作原理

### 1. 缓存键生成
缓存键由五部分组成：
- **key**: 节点名称（如 `button.submit`）
- **language**: 语言代码（如 `zh-CN`、`en-US`）
- **text**: 实际文本内容（如 `提交`、`Submit`）
- **fontFamily**: 字体族（如 `Noto Sans S Chinese`）
- **fontSize**: 字号（如 `12`）

缓存键格式：`key|language|text|fontFamily|fontSize`

例如：`button.submit|zh-CN|提交|Noto Sans S Chinese|12`

**注意**：即使文本内容相同，如果字号或字体不同，也会被分别缓存和检测，确保在不同字体样式场景下的准确性。

### 2. 缓存生命周期

```
启动检测 → 清空缓存 → 检测文本 → 记录缓存 → 检测完成
                    ↓
            遇到重复key时跳过检测
                    ↓
            下次检测时重新清空缓存
```

**重要特性**：
- ✅ 每次检测开始时，自动清空上一次的缓存
- ✅ 缓存仅在单次检测内生效
- ✅ 确保每次新的检测都是完整的，不会遗漏任何文本

### 3. 缓存机制

#### 第一次遇到某个 key+language+text+fontFamily+fontSize 组合：
1. 检查缓存 → 未命中
2. 执行文本宽度检测
3. 记录检测结果（valid 或 overflow）
4. 添加到缓存池，同时存储容器宽度信息

#### 再次遇到相同的 key+language+text+fontFamily+fontSize 组合：
1. 检查缓存 → 找到相同键
2. 检查容器宽度差异：
   - **差异 ≤ 30%**：缓存命中 ✓，跳过检测
   - **差异 > 30%**：缓存失效，重新测量并更新缓存
3. 如果缓存命中：
   - 跳过检测（不执行宽度计算）
   - **不记录结果**（避免重复记录）
   - 输出日志：`[语言] "文本" - ⚡ 已缓存，跳过检测`
4. 如果缓存失效：
   - 输出日志：`[缓存检查] 容器宽度差异 X% > 30%，需要重新测量并更新缓存`
   - 执行文本宽度检测
   - 更新缓存中的容器宽度信息

## 使用场景

### 典型场景1：Figma设计稿中的重复组件

如果在 Figma 文件中，同一个按钮组件被复制多次使用：

```
Component 1: button.submit → 提交
Component 2: button.submit → 提交  (重复)
Component 3: button.submit → 提交  (重复)
Component 4: button.cancel → 取消
```

检测日志：
```
[zh-CN] "提交" - 执行检测
[zh-CN] "提交" - ⚡ 已缓存，跳过检测
[zh-CN] "提交" - ⚡ 已缓存，跳过检测
[zh-CN] "取消" - 执行检测
```

结果：只检测了 2 个文本，跳过了 2 个重复文本

### 典型场景2：多语言翻译中的重复文本

CSV翻译文件中，不同的 key 可能指向相同的翻译：

```csv
key,zh-CN,en-US
button.submit,提交,Submit
button.confirm,提交,Confirm
button.ok,确定,OK
```

如果两个不同的 key 都翻译成"提交"，它们不会被缓存（因为 key 不同）：
- `button.submit|zh-CN|提交|Noto Sans S Chinese|12` → 检测
- `button.confirm|zh-CN|提交|Noto Sans S Chinese|12` → 检测（key不同，不会命中缓存）

**注意**：只有完全相同的 key+language+text+fontFamily+fontSize 组合才会命中缓存。此外，即使缓存键相同，如果容器宽度差异超过 30%，也会重新测量。

## 缓存统计

每次检测完成后，会输出详细的缓存统计信息：

```javascript
{
  total: 10,              // 总请求次数
  hits: 3,                // 缓存命中次数
  misses: 7,              // 缓存未命中次数（实际检测次数）
  cacheSize: 7,           // 缓存池大小
  hitRate: "30.00%"       // 命中率
}
```

### 统计指标说明

- **total**: 所有翻译文本的总数量（包括重复的）
- **hits**: 遇到重复文本且容器宽度差异在容忍范围内的次数（跳过检测）
- **misses**: 实际执行检测的次数（包括首次检测和容器宽度差异过大导致的重新检测）
- **cacheSize**: 唯一的 key+language+text+fontFamily+fontSize 组合数量
- **hitRate**: 命中率 = (hits / total) × 100%

## API 使用

### 自动使用（推荐）

缓存功能在 `validateI18nTexts` 方法中自动启用，无需额外配置：

```javascript
const TextWidthChecker = require('./textWidthChecker');
const checker = new TextWidthChecker();

// 每次调用都会自动清空缓存并启用缓存机制
const results = checker.validateI18nTexts(textNodes, translations);

// 查看缓存统计
console.log(results.cacheStats);
```

### 手动管理（高级用法）

如果需要手动管理缓存：

```javascript
// 检查是否在缓存中（需要提供完整的参数，包括字体样式和容器宽度）
const inCache = checker.isInCache(
  'button.submit', 
  'zh-CN', 
  '提交',
  'Noto Sans S Chinese',
  12,
  100  // containerWidth
);

// 手动添加到缓存
checker.addToCache(
  'button.submit', 
  'zh-CN', 
  '提交',
  'Noto Sans S Chinese',
  12,
  100  // containerWidth
);

// 手动清空缓存
checker.clearCache();

// 获取缓存统计
const stats = checker.getCacheStats();
```

## 性能优势

### 计算复杂度
- **无缓存**: O(n × m × c)
  - n: 节点数量
  - m: 语言数量
  - c: Canvas 测量复杂度（包括字体加载、文本测量等）

- **有缓存**: O(u × m × c)
  - u: 唯一节点数量（u ≤ n）
  - 当存在重复节点时，u < n，性能提升明显

### 实际收益
假设有 100 个文本节点，其中 30 个是重复的：
- **无缓存**: 检测 100 次
- **有缓存**: 检测 70 次 + 跳过 30 次
- **性能提升**: 30%

## 注意事项

### 1. 缓存范围
- ✅ 缓存仅在单次检测内生效
- ✅ 下次检测开始时会自动清空
- ❌ 缓存不会跨检测持久化

### 2. 结果记录
- 已缓存的文本**不会**被记录到 `valid` 或 `overflow` 数组中
- 这样可以避免重复记录相同的问题
- 结果中的 `summary.total` 只计算实际检测的数量，不包括缓存命中的数量

### 3. 缓存键的唯一性
缓存键由五部分组成，必须**完全一致**才能命中缓存：
```javascript
// 这些会被视为不同的缓存项：
'button.submit|zh-CN|提交|Noto Sans S Chinese|12'      // key不同
'button.cancel|zh-CN|提交|Noto Sans S Chinese|12'      // key不同，即使文本相同

'button.submit|zh-CN|提交|Noto Sans S Chinese|12'      // language不同
'button.submit|en-US|提交|Noto Sans S Chinese|12'      // language不同

'button.submit|zh-CN|提交|Noto Sans S Chinese|12'      // text不同
'button.submit|zh-CN|提交按钮|Noto Sans S Chinese|12'  // text不同

'button.submit|zh-CN|提交|Noto Sans S Chinese|12'      // fontSize不同
'button.submit|zh-CN|提交|Noto Sans S Chinese|14'      // fontSize不同

'button.submit|zh-CN|提交|Noto Sans S Chinese|12'      // fontFamily不同
'button.submit|zh-CN|提交|Arial|12'                   // fontFamily不同
```

### 4. 容器宽度模糊匹配
即使缓存键完全相同，系统还会检查容器宽度：
- **差异 ≤ 30%**：使用缓存结果，跳过检测
- **差异 > 30%**：缓存失效，重新测量并更新缓存中的容器宽度

示例：
```javascript
// 第一次检测：容器宽度 100px
'button.submit|zh-CN|提交|Noto Sans S Chinese|12' → 检测，缓存 containerWidth: 100

// 第二次检测：容器宽度 120px（差异 20%）
'button.submit|zh-CN|提交|Noto Sans S Chinese|12' → 缓存命中 ✓（差异 < 30%）

// 第三次检测：容器宽度 150px（差异 50%）
'button.submit|zh-CN|提交|Noto Sans S Chinese|12' → 缓存失效，重新检测，更新缓存 containerWidth: 150
```

## 最新更新 (v1.1.0)

### 新增功能

1. **字体样式缓存键**
   - 缓存键现在包含 `fontFamily` 和 `fontSize`
   - 确保不同字体样式的文本被正确区分和检测
   - 提升在不同模块中字体样式不同的场景下的准确性

2. **容器宽度模糊匹配**
   - 缓存存储结构从 `Set` 升级为 `Map`，可存储容器宽度信息
   - 自动检查容器宽度差异，容忍 30% 的差异范围
   - 在保证准确性的同时，提升缓存命中率

### 技术改进

- 缓存存储：`Set<string>` → `Map<string, {containerWidth: number}>`
- 缓存键组成：3 部分 → 5 部分（新增 fontFamily, fontSize）
- 智能宽度检查：新增 30% 差异容忍范围

## 总结

缓存池功能的核心优势：

1. **提高效率**：避免重复检测相同的文本，支持容器宽度模糊匹配
2. **自动管理**：每次检测自动清空，无需手动干预
3. **准确性保证**：考虑字体样式和容器宽度，确保每次检测都是完整的，不会遗漏
4. **透明统计**：提供详细的缓存命中率和性能数据
5. **简化结果**：自动过滤重复记录，结果更清晰
6. **智能匹配**：容器宽度差异容忍，提升缓存命中率

通过这个功能，在处理大型 Figma 文件或有大量重复组件的设计稿时，可以显著提升检测速度和效率。特别是在不同模块中字体样式不同、容器宽度略有差异的场景下，缓存机制能够智能处理，既保证准确性，又提升性能。
